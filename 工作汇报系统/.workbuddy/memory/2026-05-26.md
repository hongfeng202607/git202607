## 前端 axios 响应解包 Bug

前端 AI 配置页测试连接一直显示"连接失败"，但后端 curl 测试连接成功。

### 根因
axios 响应拦截器（api/index.js 第15行 `response => response.data`）已经解包了 `response.data`，
所以 `api.post()` 返回的直接就是后端 JSON body（如 `{code:200, msg:"连接成功"}`）。

但 `testConnection` 和 `generateByAI` 写的是：
```js
const resp = await api.post(...)  // resp 已经是 {code:200, msg:"连接成功"}
const d = resp.data || {}         // d = undefined || {} = {}  ← BUG！
d.code === 200                    // undefined === 200 → false → 显示失败
```

### 修复
- `testConnection`：改为 `const d = await api.post(...)`，直接用 `d.code` 和 `d.msg`
- `generateByAI`：同上，改为 `const d = await api.post(...)`，用 `d.code` 和 `d.data?.content`
- 其他页面（loadConfig/loadUsers 等）用 `r.data` 取后端 JSON 的 `data` 字段，这是正确的，不需要改

## 周期汇报锁定范围修正（已通过不再锁定）

红枫反馈：新增的工作记录（还没生成周期汇报），点"保存并提交"后撤回按钮变灰。

### 根因
数据库有两条测试时创建的月报（report_id 12/13），report_status=3（已通过），is_delete=0，日期范围覆盖整个5月。
`checkCycleReportLocked` 检查 `report_status IN (1,2,3)`，导致已通过的汇报也锁定工作记录。

### 逻辑分析
- 已提交(1)的汇报：接收人正在审批中，撤回记录会影响汇报内容 → 应锁定 ✅
- 已退回(2)的汇报：需要修改后重新提交 → 应锁定 ✅
- 已通过(3)的汇报：审批已完成，汇报内容已确定，撤回记录不影响已通过的汇报 → 不应锁定 ❌

### 修复
1. `checkCycleReportLocked()`：`report_status IN (1,2,3)` → `IN (1,2)`
2. `GET /records` 附加 cycle_report_status 的查询：`IN (1,2,3)` → `IN (1,2)`
3. 前端锁定提示文字："周期汇报已提交，不可XX" → "周期汇报审批中，不可XX"

## 工作记录关联方案（report_record 关联表）

日期范围匹配无法区分"汇报提交前已有的记录"和"后来新增的记录"，改为关联表方案。

### 数据库变更
1. **work_record 新增 display_id 列**（VARCHAR(20)）：用户可见编号，格式 WR-0001
   - 已有记录已回填：`UPDATE SET display_id = 'WR-' || LPAD(record_id::text, 4, '0')`
   - 新建记录在 INSERT 后立即 UPDATE 生成
2. **新建 report_record 关联表**：
   - `id SERIAL PRIMARY KEY`
   - `report_id BIGINT REFERENCES cycle_report(report_id)`
   - `record_id BIGINT REFERENCES work_record(record_id)`
   - `UNIQUE(report_id, record_id)`
3. 已有周期汇报已按日期范围回填关联（8条记录）

### 后端变更
1. **record.js**：
   - `checkCycleReportLocked(userId, recordId)` → 改为查 `report_record JOIN cycle_report`，参数从 recordDate 改为 recordId
   - `GET /records`：用 `report_record JOIN cycle_report WHERE record_id = ANY($1) AND report_status IN (1,2)` 查关联
   - `PUT/DELETE /records/:id`、`POST /records/recall`：传 record_id 而非 record_date 给 checkCycleReportLocked
2. **report.js**：
   - `POST /reports`：接受 `record_ids` 数组，创建汇报时写入 report_record 关联
   - `POST /reports/submit`：提交草稿时自动关联该周期内的已提交工作记录
   - `GET /reports/work-records`：返回 record_id 和 display_id
3. **audit.js**：
   - `POST /audit/resubmit`：重新提交时删除旧关联，重新写入当前周期内已提交记录的关联

### 前端变更
1. **Records.vue**：表格新增"编号"列（display_id）
2. **Reports.vue**：
   - 新建/编辑汇报时，工作记录列表改为 checkbox 可勾选，加载后默认全选
   - 提交时传 `record_ids` 数组
   - 显示 display_id + 日期 + 内容摘要

### 关键排坑经验
- Windows Git Bash 下 `taskkill //F //IM node.exe` 不一定杀死所有 node 进程，应用 PowerShell 的 `Get-Process -Name node | Stop-Process -Force`
- Express 5.2.1 路由参数 `:id` 后接后缀（如 `/:id/edit`）可能不匹配，改用 `:id` 作为最后一个路径段或将 edit 前置（`/edit/:id`）
- JWT 中 BIGINT user_id 被序列化为字符串，`parseInt("1") === "1"` 因类型不同返回 false，需用 `String() === String()` 或 `Number() === Number()` 比较
